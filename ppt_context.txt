SECTION 1: PROJECT OVERVIEW
================================================================================

TITLE: Structural Claim Analytics for Infringement Prediction

OBJECTIVE:
Develop a machine learning system to predict patent infringement risk by
analyzing structural and semantic features of patent claims against product
descriptions.

KEY COMPONENTS:
- ML Model: XGBoost Regressor with isotonic calibration
- Feature Engineering: 45+ features capturing claim structure, language patterns,
  domain overlap, and semantic similarity
- Explainability: SHAP (SHapley Additive exPlanations) for feature importance
- Frontend: Streamlit web application for interactive analysis

================================================================================
SECTION 2: TECHNICAL ARCHITECTURE
================================================================================

TECHNOLOGY STACK:
- Backend: Python (scikit-learn, XGBoost, spaCy, SentenceTransformers)
- Frontend: Streamlit
- ML Framework: XGBoost with sklearn Pipeline
- NLP: spaCy (en_core_web_sm), Sentence-BERT (all-MiniLM-L6-v2)
- Explainability: SHAP (TreeExplainer)

PIPELINE:
Input (Claim + Product) → FeatureBuilder → Feature Vector (45 features)
→ XGBoost Pipeline (StandardScaler → XGBRegressor) → Isotonic Calibration
→ Risk Score (0-100%)

OUT-OF-DOMAIN VOCABULARIES (for true negatives):
- pharmaceutical, construction, ecommerce, finance, marketing, food, textiles, agriculture

CLAIM FEATURES (from claim_text):
1. claim_word_count - number of words
2. claim_token_count - number of tokens (spaCy)
3. claim_clause_markers - count of "wherein", "whereby", "such that", etc.
4. claim_dep_depth_max - maximum dependency parsing tree depth
5. claim_dep_depth_mean - mean dependency tree depth
6. claim_func_verb_density - ratio of functional verbs (transmit, process, detect, etc.)
7. claim_struct_noun_density - ratio of structural nouns (server, processor, etc.)
8. claim_abstraction_score - count of abstraction patterns
9. claim_means_plus_function - binary flag for "means for" language
10. claim_interaction_density - density of interaction terms (communicate, transmit, etc.)
11. claim_unique_nouns - count of unique structural nouns
12. claim_abstractness - computed abstractness (0=concrete, 1=abstract)
13. claim_domain_score - confidence of domain detection

PRODUCT FEATURES (from product_text):
1. prod_token_count - number of tokens
2. prod_component_terms - count of structural nouns
3. prod_interaction_terms - count of interaction verbs
4. prod_functional_terms - count of functional verbs
5. prod_functional_density - ratio of functional terms
6. prod_unique_nouns - count of unique structural nouns
7. product_domain_score - confidence of domain detection

CROSS-PAIR FEATURES:
1. semantic_similarity - Sentence-BERT cosine similarity (0-1)
2. same_domain - binary flag (1 if both detected same domain)
3. domain_overlap_count - number of overlapping domains
4. any_domain_overlap - binary for any domain match
5. domain_mismatch_penalty - penalty when domains don't match
6. neither_domain_detected - flag for non-technical texts
7. short_neither_penalty - penalty for short generic texts
8. noun_jaccard - Jaccard similarity of structural nouns
9. noun_overlap_count - count of matching structural nouns
10. noun_overlap_claim - ratio of claim nouns found in product
11. noun_overlap_prod - ratio of product nouns found in claim
12. all_noun_jaccard - Jaccard of all nouns (not just structural)
13. bigram_jaccard - Jaccard of 2-word ngrams
14. trigram_jaccard - Jaccard of 3-word ngrams
15. len_ratio - claim_length / product_length
16. len_diff_norm - normalized length difference
17. abs_alignment - alignment of abstraction levels
18. abs_diff - absolute abstraction difference
19. complexity_balance - claim_complexity - product_detail
20. ood_keyword_ratio - ratio of out-of-domain keywords

SECTION 4: SHAP EXPLAINER - HOW IT WORKS
================================================================================

WHAT IS SHAP?
SHAP (SHapley Additive exPlanations) is a game-theoretic approach to explain
the output of any machine learning model. It computes the contribution of each
feature to the prediction.

HOW IT WORKS IN THIS PROJECT:

1. BASE VALUE (Expected Value):
   - The average model output over the training dataset
   - Represents the starting point before any features are considered
   - For XGBoost: regressor.base_score (typically ~50 for risk scores)
   - Fallback: 50.0 if base_score not available

2. SHAP VALUES COMPUTATION:
   - Uses TreeExplainer for XGBoost models (optimized for tree-based models)
   - Computed as: explainer.shap_values(X_scaled)[0]
   - Each feature gets a SHAP value: positive (increases score) or negative
     (decreases score)
   - Sum of all SHAP values + base_value = final prediction

3. FEATURE IMPORTANCE:
   - Top features are sorted by absolute SHAP value magnitude
   - Shows which features had the biggest impact on the prediction
   - Displayed as bar charts and raw values

4. FEATURE GROUPS:
   The 45 features are grouped into 5 categories for summary:

   "Claim -- structure":
   - claim_word_count, claim_token_count, claim_dep_depth_max, claim_dep_depth_mean

   "Claim -- language":
   - claim_clause_markers, claim_abstraction_score, claim_means_plus_function,
   - claim_func_verb_density, claim_struct_noun_density, claim_interaction_density

   "Product -- structure":
   - prod_token_count, prod_component_terms, prod_functional_terms

   "Product -- language":
   - prod_interaction_terms, prod_functional_density

   "Cross-pair similarity":
   - semantic_similarity

5. GROUPED CONTRIBUTION:
   - Sums absolute SHAP values within each group
   - Shows which aspect of the claim-product pair most influenced the score
   - Useful for high-level interpretation

SHAP EXPLANATION OUTPUT:
- base_value: Starting prediction value
- prediction: Final calibrated risk score
- shap_values: Dict of top 8 features with individual contributions
- grouped_shap: Dict of 5 groups with summed contributions

INTERPRETATION:
- Positive SHAP value → pushes score higher (increases risk)
- Negative SHAP value → pushes score lower (decreases risk)
- Large absolute value → strong influence on prediction
- Example: semantic_similarity=+3.2 means this feature increased risk by 3.2%

SECTION 5: MODEL TRAINING DETAILS
================================================================================

MODEL ARCHITECTURE:
- Pipeline: StandardScaler → XGBRegressor
- XGBoost Parameters:
  - n_estimators: 500
  - learning_rate: 0.05
  - max_depth: 7
  - subsample: 0.9
  - colsample_bytree: 0.9
  - reg_lambda: 1.0

TRAINING PROCESS:
1. Data loaded from synthetic_pairs_5k.csv (5000 samples)
2. Features extracted using FeatureBuilder
3. 70-30 train-test split (random_state=42)
4. 5-Fold Cross-Validation on training set
5. XGBoost trained on full training set
6. Isotonic Regression calibration on held-out portion

CALIBRATION:
- IsotonicRegression (y_min=0, y_max=100, out_of_bounds="clip")
- Fitted on out-of-sample predictions to correct systematic bias
- Final model selected based on MAE comparison

MODEL PERFORMANCE (from train_report.json):
- Cross-Validation RMSE: 16.55 (+/- 0.35)
- Cross-Validation R2: 0.728 (+/- 0.019)
- Cross-Validation MAE: 11.04 (+/- 0.26)
- Test MAE: 11.33
- Test RMSE: 17.53
- Test R2: 0.702
- Calibrator: isotonic

PREDICTION DISTRIBUTION:
- 0-20%: 734 predictions
- 21-50%: 151 predictions
- 51-75%: 486 predictions
- 76-100%: 129 predictions

SECTION 9: KEY CONCEPTS TO EXPLAIN
================================================================================

1. WHY STRUCTURAL FEATURES?
   - Patent claims define scope through structure (components, connections)
   - Structural noun overlap indicates potential claim coverage
   - Different domains have different typical structures

2. WHY SEMANTIC SIMILARITY?
   - Sentence-BERT captures meaning beyond exact words
   - Can detect similar functionality even with different wording
   - Complements structural matching

3. WHY DOMAIN DETECTION?
   - Same-domain claims/products more likely to infringe
   - Cross-domain may have partial overlap
   - Out-of-domain indicates no infringement risk

4. WHY ABSTRACTION ALIGNMENT?
   - Abstract claims (legal language) need detailed products for coverage
   - Concrete claims (specific components) need matching structure
   - Misaligned abstraction levels indicate different scope expectations

5. WHY SHAP?
   - Black-box model needs explanation for legal/compliance use
   - Shows WHY a score was given, not just WHAT the score is
   - Enables trust and verification by domain experts
